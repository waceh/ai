### 핵심 답변

**결론부터 말하면, 제품이 다르면 보통 결과도 다릅니다.** Claude(챗), Codex(코딩 에이전트), Cursor Agent처럼 이름이 달라 보이는 이유는, 겉에 **LLM**만 있는 게 아니라 **Prompt·도구·맥락·Harness**가 제품마다 다르게 붙어 있기 때문입니다.

이론적으로는 **같은 모델 ID**, **같은 `messages`**, **같은 `temperature`·`max_tokens`**로 API를 직접 호출하면 생성 문장이 매우 비슷해질 수 있습니다. 하지만 우리가 일상에서 쓰는 제품 UI는 그 조건을 맞춰 주지 않습니다. 시스템 지시, 열린 파일, 대화 기록, 사용 가능한 Tool 목록이 자동으로 달라지기 때문에 **체감 답변은 거의 항상 다릅니다.**

### "같은 모델"이란 무엇인가

여기서 말하는 "같다"는 **UI에 보이는 제품 이름**이 아니라, API 요청에 들어가는 **모델 ID 문자열**이 같다는 뜻입니다.

- 예: `claude-sonnet-4-20250514`처럼 벤더가 정한 정확한 이름
- "Sonnet", "Claude 4" 같은 **마케팅 이름**만 같다고 해서 백엔드가 동일하다고 보장되지는 않습니다
- 제품·리전·시점에 따라 **실제로는 다른 버전**이 라우팅될 수 있습니다

같은 모델인지 확인하려면 개발자 도구, API 로그, Observability trace에서 `model` 필드를 직접 비교하는 것이 가장 확실합니다.

### 무엇이 같아야 "비슷한" 결과인가

아래가 **전부** 같을 때, 순수 LLM 생성만 놓고 보면 가장 가깝습니다.

| 항목 | 설명 |
|------|------|
| 모델 ID | 완전히 동일한 문자열 |
| messages | system + user + assistant 히스토리 내용 동일 |
| 샘플링 | `temperature`, `top_p` 등 동일 |
| 부가 입력 | Tool 정의, RAG 청크, 이미지 유무 동일 |
| Tool 실행 결과 | Agent가 아닌 **단일 completion** 호출 |

실제 제품은 위 표의 오른쪽 열(맥락·Tool·Harness)이 제품마다 다르게 채워집니다. 그래서 "같은 질문을 했는데"라고 느껴도 **모델에 들어간 전체 입력**은 다를 수밖에 없습니다.

### 제품마다 결과가 달라지는 이유 (구체적으로)

1. **System Prompt·정책**
   - 챗 앱: 짧고 친절하게, 안전 정책 강조.
   - IDE Agent: 코드 스타일, 프로젝트 규칙, diff 형식으로 답하라는 지시.
   - 같은 User Prompt라도 **역할 정의**가 다르면 문장 길이·구조·톤이 바뀝니다.

2. **맥락 주입 (가장 흔한 차이)**
   - **Cursor**: 현재 열린 파일, 커서 위치, `@`로 지정한 심볼, `.cursor/rules` 등이 **Prompt**에 붙습니다.
   - **Codex / 코딩 에이전트**: 리포지토리 트리, PR, CI 로그, 터미널 출력이 붙을 수 있습니다.
   - **Claude 웹/API**: 대화 히스토리·첨부 문서 위주, IDE 파일 트리는 기본적으로 없습니다.
   - **Memory**·**RAG** 소스가 다르면 모델이 참고하는 "사실" 자체가 달라집니다.

3. **Tool Use·Skills·MCP**
   - Agent 제품은 "바로 답하기" vs "파일 읽고 → 검색하고 → 수정하기" 경로가 다릅니다.
   - Cursor는 파일 편집·터미널·MCP 서버, Codex는 GitHub·샌드박스 실행 등 **도구 세트**가 다릅니다.
   - 모델이 Tool을 쓰면 **중간 LLM 호출**이 추가되고, 최종 답은 Tool 결과에 크게 좌우됩니다.

4. **Harness (실행·안전·재시도)**
   - **Sandbox**에서만 명령 실행, **Guardrails**로 위험 호출 차단, **HITL** 승인 후 진행.
   - 재시도·폴백 모델·타임아웃 정책이 다르면 "같은 질문"도 다른 코드·다른 파일이 수정될 수 있습니다.

5. **라우팅·버전**
   - UI에는 "Sonnet"만 보여도 실제 요청은 다른 스냅샷·리전·폴백 모델로 갈 수 있습니다.
   - 트래픽·플랜·A/B 테스트로 백엔드가 바뀌는 경우도 있습니다.

### 같은 질문, 다른 제품 — 예시

질문: *"이 함수가 왜 NullPointerException이 나요?"*

| 제품 | 모델에 실제로 들어갈 수 있는 것 | 답이 달라지는 이유 |
|------|----------------------------------|-------------------|
| Claude 챗 | 질문 텍스트 + (선택) 붙인 파일 | 스택 트레이스·프로젝트 구조 없으면 일반론적 답 |
| Cursor Agent | 질문 + 열린 파일 전체 + 관련 심볼 | 해당 파일 기준으로 줄 번호·변수 추적 가능 |
| Codex(리포 맥락) | 질문 + repo 검색 결과 + PR/CI | 여러 파일 관계까지 볼 수 있음 |
| Raw API | 개발자가 넣은 `messages`만 | 넣은 만큼만 앎 |

**User Prompt 문장은 같아도**, 모델이 보는 **전체 입력**이 다르면 답은 달라지는 것이 정상입니다.

### Claude vs Codex vs Cursor (한눈에)

- **Claude**(챗·문서·API) — 대화·분석·글쓰기. 맥락은 대화·첨부 중심. 코딩도 가능하지만 IDE 파일 트리는 기본 제공 안 함.
- **Codex**(코딩 에이전트) — 리포지토리·PR·터미널과 **Tool Use** 루프에 최적화된 **AI Agent**.
- **Cursor Agent** — IDE 안에서 파일·심볼·규칙을 자동 주입하고, 편집·검색·MCP를 쓰는 **AI Agent**.

이름에 같은 모델 패밀리(예: Claude Sonnet)가 쓰여도, **제품 = 모델 + Prompt + Tool + Harness** 묶음이라고 보면 이해하기 쉽습니다.

### 자주 헷갈리는 점

- **"같은 모델 썼는데 왜 코드가 다르지?"** — 샘플링(`temperature` > 0)만으로도 문장·변수명이 조금씩 달라질 수 있습니다. Agent는 Tool 실행 결과까지 달라지면 차이가 더 큽니다.
- **"Claude에서 잘 됐는데 Cursor에서는 안 돼"** — System Prompt, 규칙 파일, 주입 맥락, Tool 권한을 먼저 의심하세요. 모델 탓만 하기 어렵습니다.
- **"API로는 되는데 제품 UI에서는 안 됨"** — UI가 맥락·도구·길이 제한을 더 강하게 걸거나, 다른 모델로 라우팅했을 수 있습니다.
- **완전 동일 출력 기대** — 연구·회귀 테스트가 아니라면 "비슷한 품질"을 목표로 하는 편이 현실적입니다.

### 실무 체크리스트

1. **같은 모델인가?** — 로그의 `model` ID 문자열을 제품 간 diff.
2. **같은 입력인가?** — system, user, tool 목록, RAG 청크, 이미지를 diff. User 한 줄만 같다고 입력이 같지 않습니다.
3. **Agent 루프인가?** — Tool 호출 횟수·중간 `tool_result`가 있으면 단순 채팅과 비교하면 안 됩니다.
4. **제품 간 비교할 때** — 질문·파일·규칙·도구를 고정한 **Evaluation** 시나리오를 만들고, 한 변수씩만 바꿔 테스트하세요.
5. **품질 이슈 보고할 때** — "모델이 이상하다"보다 "어떤 맥락·어떤 Tool이 붙었는지"를 함께 적으면 재현이 빨라집니다.
